# tree-sitter grammar compatibility survey

Which `tree-sitter-*` grammar crates exist on crates.io, whether they are
compatible with the runtime this workspace pins, and roughly how much each would
add to the release binary. Generated by querying the crates.io API (survey date
2026-06-16); re-run the checks (see bottom) to refresh.

- **Released** — publish date of the listed (latest) version; a stale date is a
  signal the grammar may be unmaintained.
- **≈ Binary** — estimated release-binary growth from adding the grammar (see
  [Estimating binary growth](#estimating-binary-growth)). Add ~10% for linker
  glue; values are `cc -O2` compiled parser+scanner object size.
- **†** on a crate marks a **first-party** grammar (org `tree-sitter/`); the rest
  are community grammars.

The table is sorted alphabetically by language.

## How compatibility is decided

The workspace pins (see `Cargo.toml`):

| crate | version |
|---|---|
| `tree-sitter` (runtime) | `0.26.9` |
| `tree-sitter-language` (shim) | `0.1.7` |

`tree-sitter` is a `links = "tree-sitter"` native crate, so **only one version of
it may exist in a build**. That makes the deciding factor *how a grammar crate
depends on the runtime*:

- **✅ COMPAT** — depends on **`tree-sitter-language` (`^0.1`)**, not on
  `tree-sitter` directly. This is the modern, ABI-decoupled style: the crate
  exports `LANGUAGE: LanguageFn` and links against whatever `tree-sitter`
  runtime the workspace provides (0.26.9 here). Wires exactly like the four
  grammars already in the tree.
- **❌ INCOMPAT** — depends **directly on an old `tree-sitter`** (e.g. `^0.20`,
  `>=0.21,<0.23`). Cargo cannot satisfy that alongside `0.26.9` (single-version
  `links` crate), so it won't build. Usually a maintained `*-ng` fork exists —
  use that.
- **⚠️ CHECK** — published but no dependency metadata (placeholder/empty crate);
  inspect before relying on it.
- **— MISSING** — no such crate on crates.io.

> A COMPAT grammar still needs a thin plugin (`mod.rs` + `config.toml` +
> `dialect.rs`, or reuse of a shared dialect) per `crates/code-ranker-plugins/src/languages/README.md`.
> "Compatible" here means only *the crate will link against our runtime*.

## Estimating binary growth

A grammar's binary footprint is almost entirely its generated `parser.c` parse
tables (`const` data, landing in `__TEXT,__const`) plus a small `scanner.c`; the
Rust glue is negligible. So the growth is estimated **without building the whole
project**: download the crate, compile its `parser.c` (+ scanner) with the same
`cc -O2`, and sum the object's `__TEXT`+`__DATA` size. The **≈ Binary** column is
that figure.

**Validation.** Against the real 16.5 MB release binary (`cargo build --release`):
the four wired grammars' object sizes (TypeScript 2.7, Rust 1.1, Python 0.44,
JavaScript 0.40 MB → ≈ 4.6 MB) line up with the binary's `__TEXT,__const` section
(6.39 MB — tables + other const data). A differential build that added
`tree-sitter-go` and force-linked it grew the release binary by **226 KB** vs the
**210 KB** predicted — the estimate runs ~5–10% low (LanguageFn glue, symbol-table
entries, alignment), hence the "+10%" rule of thumb.

**Caveats.** Measured under the default `release` profile (no LTO, no `strip`).
The shipped artifact uses `--strip` (+ a thin-LTO `[profile.dist]`), which lowers
the *total* binary size but not a grammar's marginal cost — the parse tables are
referenced `const` data and survive stripping/LTO, so the per-language deltas
above hold for the shipped build too. Numbers are macOS/mach-o; other targets
vary within ~2×. Size scales with grammar complexity (states/symbols), not
language popularity — note Verilog (17 MB), OCaml/Julia/Fortran (~6 MB) vs JSON
(5 KB) or Lua (50 KB).

## Re-running the surveys

Compatibility check: query the crates.io API for each candidate, read the latest
version's normal dependencies + publish date, classify by whether it depends on
`tree-sitter-language` (decoupled → compatible) or directly on an old
`tree-sitter`. Set a descriptive `User-Agent` (crates.io requires one) and add a
small delay between requests. Alternate names (`-ng`, `-md`, `-proto`) are worth
checking when the canonical name is INCOMPAT or MISSING.

Binary-growth estimate: download the `.crate`, `tar -xzf`, find every `parser.c`
(multi-grammar crates like `tree-sitter-typescript` ship more than one), compile
each with `cc -O2 -c parser.c -I <its dir>` plus any `scanner.c`/`scanner.cc`
(`c++` for `.cc`), and sum `size <obj>`'s `__TEXT`+`__DATA` columns.
